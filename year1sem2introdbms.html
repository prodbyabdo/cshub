Lecture 1 · DBMS Introduction & File-Based Problems
Definition of DBMS
DBMS (Database Management System) is a software system that manages, stores, and retrieves data efficiently in a structured format. Acts as a bridge between a central database and multiple clients.
  • Connects database with multiple clients (applications and users)
  • Enables efficient creation, update, and query of databases
  • Ensures data integrity, consistency, and security
  • Reduces data redundancy through centralized control
  • Supports concurrent access and automatic backups
Problems with File-Based Systems
Data RedundancyDuplicate entries across files
InconsistencyConflicting or outdated information
Difficult AccessManual file search required
Poor SecurityNo control over data access
Single-User AccessNo collaboration support
No Backup/RecoveryData loss was permanent
Components of DBMS Applications

1. Hardware

Physical devices like servers, disks, input-output devices. Stores and processes data.

2. Software

DBMS software (MySQL, Oracle, PostgreSQL), OS, network software, application tools.

3. Data

Operational Data: Actual user data. Metadata: Data about data.

4. Procedures

Instructions for setup, login, validation, backup, access control, reporting.

5. Database Access Language

DDL CREATE, ALTER, DROP · DML INSERT, UPDATE, DELETE

6. People

DBAs manage security. Developers build apps. End Users use the system.

Lectures 2 · Types of DBMS
1. Relational DBMS
  • Organizes data into tables (relations)
  • Uses SQL for querying
  • Primary and foreign keys for relationships

Examples: MySQLOraclePostgreSQL

2. NoSQL DBMS
  • Handles large-scale data
  • Non-relational formats (documents, key-value)
  • Flexible schemas for unstructured data

Examples: MongoDBRedis

3. Object-Oriented DBMS
  • Integrates OOP concepts
  • Stores data as objects
  • Ideal for complex data modeling

Examples: ObjectDBdb4o

Lectures 3 & 4 · Database Languages

DDL — Data Definition Language

Deals with database schemas and structure definitions.

CREATE Create database objects
ALTER Modify existing structure
DROP Delete objects
TRUNCATE Remove all records

DML — Data Manipulation Language

Focuses on manipulating stored data.

INSERT Insert data
UPDATE Update existing data
DELETE Delete records
MERGE UPSERT operation

DCL — Data Control Language

Manages access permissions and security.

GRANT Give permissions
REVOKE Remove permissions

TCL — Transaction Control Language

Oversees transaction management.

COMMIT Save changes
ROLLBACK Undo changes
SAVEPOINT Set rollback point
Lecture 5 · Database Abstraction & Levels
Three Levels of Abstraction
External
(View)
Highest abstraction. User-specific data views customized by needs and permissions.
Conceptual
(Logical)
Defines what data is stored and relationships. Independent of physical storage.
Internal
(Physical)
Lowest level. Describes actual storage on hardware (files, indexes, access paths).
Data Independence

Changes at one abstraction level should not affect others.

Physical Independence

Storage changes don't affect logical schema.

Logical Independence

Logical schema changes don't affect external views.

Primary & Foreign Keys
🔑 Primary Key (PK): Uniquely identifies each record. Cannot be NULL.
🔗 Foreign Key (FK): References a Primary Key in another table.

Short-Answer Key (20)

  1. DBMS Definition: Software for creating and managing databases. Advantages: reduces data redundancy, improves security and data sharing.
  2. File System vs DBMS: File systems store isolated data with duplication risk. DBMS provides centralized control, querying, concurrency, and integrity constraints.
  3. Primary vs Foreign Key: Primary key uniquely identifies each record in its table. Foreign key references a primary key in another table for relationships.
  4. CREATE TABLE Student (RollNo INT PRIMARY KEY, Name VARCHAR(50), Department VARCHAR(30));
  5. Normalization: Organizing data to reduce redundancy and update anomalies. Required for consistency and maintainability.
  6. Normal Forms: 1NF uses atomic fields; 2NF removes partial dependency on composite keys; 3NF removes transitive dependency on non-key attributes.
  7. ACID: Atomicity (all-or-nothing), Consistency (valid state), Isolation (independent transactions), Durability (persistent changes).
  8. SQL Languages: DDL: CREATE; DML: UPDATE; DQL: SELECT; DCL: GRANT; TCL: COMMIT.
  9. Schema vs Instance: Schema is the structural blueprint. Instance is the current data state at a specific time.
  10. Joins: INNER JOIN returns matching rows only. LEFT JOIN returns all left table rows plus matching right table rows.
  11. Index: Fast lookup structure on columns. Benefit: faster searches. Drawback: extra storage, slower writes.
  12. Abstraction: Hides complexity across three levels: physical (hardware), logical/conceptual (schema), and view (user-specific).
  13. Data Integrity: Ensures correctness and consistency. Entity integrity: non-null, unique primary keys. Referential integrity: valid foreign key references.
  14. Centralized vs Distributed: Centralized: single location management. Distributed: data spread across multiple sites with coordination.
  15. View: Virtual table based on a query. Simplifies reporting and restricts sensitive data exposure.
  16. INSERT INTO Student (RollNo, Name, Department) VALUES (101, 'Amina', 'CS');
  17. UPDATE Student SET Department = 'IT' WHERE RollNo = 101;
  18. SELECT Department, COUNT(*) AS total_students FROM Student GROUP BY Department;
  19. Backup & Recovery: Backup creates restorable data copies. Recovery restores consistency after failure.
  20. Concurrency Control: Coordinates simultaneous transactions to avoid conflicts like lost updates and dirty reads.
